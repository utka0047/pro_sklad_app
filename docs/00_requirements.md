## 2. Назначение и цели системы

### 2.1 Назначение

Приложение предназначено для автоматизации складского учёта на предприятии. Оно обеспечивает сотрудникам склада возможность в реальном времени фиксировать приход и расход товаров, проводить инвентаризацию, отслеживать остатки и работать со штрихкодами товаров с мобильного устройства.

### 2.2 Проблемы, которые решает система

- Отказ от бумажного учёта движений товаров
- Устранение расхождений между фактическими и учётными остатками
- Оперативный доступ к информации о складе с мобильного устройства
- Ускорение процесса инвентаризации за счёт сканирования штрихкодов
- Своевременное оповещение о критически низких остатках

### 2.3 Целевая аудитория

- **Кладовщики** — фиксируют приход/расход, проводят инвентаризацию
- **Менеджеры склада** — контролируют остатки, просматривают аналитику
- **Директор / руководство** — получают сводную статистику по складу

## 3. Требования к функциональности

### 3.1 Главный экран (Дашборд)

Система должна отображать:

- **Сводку по складу:**
  - Общее количество наименований товаров
  - Суммарная стоимость товарных остатков (в рублях, целое число)
  - Количество позиций с критически низким остатком
  - Количество операций за текущий день
- **Список товаров с низким остатком** — позиции, у которых `currentStock ≤ minStock`
- **Кнопки быстрых действий:**
  - Перейти к сканеру штрихкодов
  - Создать операцию прихода
  - Перейти к инвентаризации
- **Кнопка ручной синхронизации** с сервером

### 3.2 Список товаров

Система должна обеспечивать:

- Отображение всех товаров в виде списка с прокруткой
- Для каждого товара в списке: наименование, артикул (SKU), текущий остаток, категория, единица измерения
- **Визуальную индикацию** — выделение красным цветом позиций с низким остатком
- **Поиск в реальном времени** по наименованию и артикулу (debounce 300 мс)
- Переход к карточке товара по нажатию

### 3.3 Карточка товара

Система должна отображать:

- Полную информацию о товаре: наименование, артикул, категория, единица измерения, цена (в рублях)
- Текущий остаток и минимально допустимый остаток
- **Историю движений** данного товара (тип, количество, дата, комментарий)
- Кнопки для создания операции прихода и расхода по данному товару

### 3.4 Операции с товарами (движения)

Система должна поддерживать четыре типа операций:

| Тип | Название | Описание |
|-----|----------|----------|
| `IN` | Приход | Поступление товара на склад — увеличивает остаток |
| `OUT` | Расход | Списание товара — уменьшает остаток |
| `TRANSFER` | Перемещение | Перемещение между местами хранения |
| `INVENTORY` | Инвентаризация | Корректировка остатка по результатам пересчёта |

Форма создания операции должна содержать:
- Выбор товара (или товар уже выбран при переходе из карточки)
- Тип операции
- Количество (целое число, > 0)
- Комментарий (необязательно)

После успешного создания операции:
- Показать уведомление (Snackbar) «Операция выполнена»
- Запустить синхронизацию данных с сервером
- Вернуться на предыдущий экран

### 3.5 Сканер штрихкодов

Система должна обеспечивать:

- Запуск камеры устройства для сканирования
- Автоматическое распознавание штрихкодов форматов: EAN-13, EAN-8, QR-код, Code 128, Code 39
- По результату сканирования — поиск товара по артикулу (SKU) и переход к его карточке
- Управление вспышкой (фонариком) во время сканирования: кнопка включения/выключения
- Корректное освобождение ресурсов камеры при выходе с экрана

### 3.6 Инвентаризация

Система должна обеспечивать:

- Загрузку полного списка товаров с текущими учётными остатками
- Возможность ввести **фактический** остаток для каждой позиции
- Отображение **отклонения** (разница между фактическим и учётным остатком) с цветовой индикацией: зелёный — совпадает или больше, красный — меньше
- Кнопку «Завершить инвентаризацию» — сохраняет все отклонения как движения типа `INVENTORY`
- После завершения — запуск синхронизации с сервером

### 3.7 Фоновая синхронизация

Система должна обеспечивать:

- **Периодическую синхронизацию** каждые 3 часа при наличии интернет-соединения
- **Событийную синхронизацию** — немедленно после каждой успешной операции (движение, инвентаризация)
- Работу синхронизации в фоне, не блокируя интерфейс
- Повтор синхронизации при ошибке (RetryPolicy)
- Сохранение задачи синхронизации при выключении устройства (восстановление после перезагрузки)
- Выполнение синхронизации только при наличии сетевого подключения

### 3.8 Офлайн-режим

Система должна обеспечивать:

- Полную работоспособность приложения без интернет-соединения (чтение данных из локальной базы)
- При отсутствии сети — отображение данных из последнего кэша без сообщений об ошибке
- При появлении сети — автоматическую синхронизацию поставленных в очередь задач

## 4. Требования к интерфейсу

### 4.1 Навигация

- **Single Activity Architecture** — одна Activity, все экраны реализованы как Fragments
- Нижняя навигационная панель (`BottomNavigationView`) с вкладками: «Главная» и «Товары»
- Навигация между экранами — через Navigation Component (Android Jetpack)

### 4.2 Поддержка безрамочных экранов (Edge-to-Edge)

Приложение должно корректно отображаться на устройствах:
- С жестовой навигацией (Android 10+)
- С аппаратными кнопками навигации
- С вырезом (notch) под фронтальную камеру

Требования:
- Контент не должен перекрываться статусной строкой
- `BottomNavigationView` должна учитывать высоту системной навигационной панели
- Реализация через `WindowCompat.setDecorFitsSystemWindows(false)` + `WindowInsetsCompat`

### 4.3 Визуальные требования

- Цветовая схема: Material Design 3
- Язык интерфейса: русский
- Денежные суммы отображаются с символом ₽ (рубль), целым числом (без копеек)
- Цветовая индикация движений: зелёный — приход (IN), красный — расход (OUT), синий — перемещение/инвентаризация
- Индикация загрузки (`ProgressBar`) при выполнении сетевых запросов
- Сообщения об ошибках через `Snackbar`

### 4.4 Адаптер списков

- `RecyclerView` с `DiffUtil` для эффективного обновления списков
- Поддержка прокрутки без мерцания при обновлении данных

## 5. Требования к архитектуре

### 5.1 Принципы

- **Clean Architecture** — строгое разделение на три слоя: Presentation, Domain, Data
- **MVVM** (Model-View-ViewModel) для слоя Presentation
- Зависимости направлены строго внутрь: Presentation → Domain ← Data

### 5.2 Слой Domain

- Содержит только чистый Kotlin-код, без зависимостей от Android SDK
- Доменные модели: `Product`, `Movement`, `WarehouseSummary`
- Интерфейсы репозиториев: `ProductRepository`, `MovementRepository`
- Sealed-класс `Result<T>` для унифицированной обработки результатов

### 5.3 Слой Data

- Реализации репозиториев следуют стратегии **offline-first**: при ошибке сети возвращают данные из локального кэша
- DTO-классы для десериализации JSON (с `@SerializedName` для snake_case полей)
- Entity-классы для хранения в Room
- Mapper-функции для преобразования: `DTO → Entity → Domain Model`

### 5.4 Слой Presentation

- Каждый экран — отдельный Fragment со своим ViewModel
- ViewModel не хранит ссылок на View
- Состояние UI представлено sealed-классом `UiState` (Loading / Success / Error)
- Реактивное обновление UI через `LiveData`

### 5.5 Внедрение зависимостей

- **Hilt** (Dagger 2) — DI-контейнер для всего приложения
- Все зависимости объявлены как `@Singleton` (один экземпляр на всё приложение)
- Hilt-модули: `NetworkModule`, `DatabaseModule`, `RepositoryModule`, `AppModule`

## 6. Требования к серверной части

### 6.1 Общие требования

- REST API с форматом обмена данными JSON
- Базовый URL: задаётся через BuildConfig (возможность смены без пересборки)
- Все эндпоинты поддерживают пагинацию (`skip`, `limit`)

### 6.2 Обязательные эндпоинты

**Товары:**

| Метод | Путь | Описание |
|-------|------|----------|
| `GET` | `/products/` | Список товаров |
| `GET` | `/products/{id}` | Карточка товара |
| `GET` | `/products/categories` | Список категорий |
| `POST` | `/products/` | Создать товар |
| `PUT` | `/products/{id}` | Обновить товар |

**Движения:**

| Метод | Путь | Описание |
|-------|------|----------|
| `GET` | `/movements/` | Список движений (с фильтрами) |
| `POST` | `/movements/` | Создать движение |

**Аналитика:**

| Метод | Путь | Описание |
|-------|------|----------|
| `GET` | `/analytics/summary` | Сводка по складу |
| `GET` | `/analytics/low-stock` | Товары с низким остатком |

### 6.3 Требования к ответам

- При создании движения сервер автоматически пересчитывает `current_stock` товара
- Числовые поля с дробной частью (стоимость остатков) возвращаются как строка
- Поля дат — строка в формате ISO 8601
- Все имена полей в snake_case (`current_stock`, `min_stock`, `movement_type`)

### 6.4 Обработка ошибок

Сервер должен возвращать стандартные HTTP-коды:
- `200 OK` — успешный запрос
- `201 Created` — ресурс создан
- `400 Bad Request` — ошибка в параметрах запроса
- `404 Not Found` — ресурс не найден
- `500 Internal Server Error` — ошибка сервера

## 7. Требования к безопасности

- Хранение базового URL API в `BuildConfig` (не хардкодить в коде)
- Разрешения Android: `INTERNET`, `CAMERA`, `ACCESS_NETWORK_STATE`, `VIBRATE`
- Разрешение на использование камеры запрашивается в runtime (при первом открытии сканера)
- Логирование HTTP-запросов (`HttpLoggingInterceptor`) только в debug-сборке
- Не хранить чувствительные данные (токены, пароли) в SharedPreferences без шифрования

|---------|------------|

## 10. Модели данных

### 10.1 Товар (Product)

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | Int | Уникальный идентификатор |
| `name` | String | Наименование товара |
| `sku` | String | Артикул (уникальный) |
| `category` | String | Категория |
| `unit` | String | Единица измерения (шт, кг, л, ...) |
| `price` | Double | Цена в рублях |
| `description` | String? | Описание (необязательно) |
| `minStock` | Int | Минимальный допустимый остаток |
| `currentStock` | Int | Текущий остаток |
| `createdAt` | String | Дата создания |
| `updatedAt` | String | Дата последнего обновления |
| `isLowStock` | Boolean | Вычисляемое: `currentStock ≤ minStock` |

### 10.2 Движение товара (Movement)

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | Int | Уникальный идентификатор |
| `productId` | Int | ID товара |
| `productName` | String | Наименование товара |
| `productSku` | String | Артикул товара |

| `quantityAfter` | Int | Остаток после операции |
| `comment` | String? | Комментарий (необязательно) |

|------|-----|----------|

## 11. Этапы разработки

### Этап 1. Подготовка и базовая структура

- Настройка проекта: Gradle, зависимости, Hilt
- Создание структуры пакетов (Clean Architecture)
- Настройка Room: Entity-классы, DAO, AppDatabase
- Настройка Retrofit: OkHttp, ApiService, DTO-классы
- Реализация Mapper-функций и sealed-класса Result
- DI-модули: NetworkModule, DatabaseModule, RepositoryModule

**Результат:** Каркас приложения с настроенными слоями, компилируется без ошибок.

---

### Этап 2. Основные экраны

- MainActivity: навигация, BottomNavigationView, edge-to-edge insets
- HomeFragment + HomeViewModel: сводка, низкие остатки, быстрые действия
- ProductListFragment + ProductListViewModel: список с поиском, адаптер
- ProductDetailFragment + ProductDetailViewModel: карточка товара, история движений

**Результат:** Пользователь может просматривать товары и их детали.

---

### Этап 3. Операции с товарами

- MovementFragment + MovementViewModel: форма создания операции, валидация
- Типы операций IN / OUT / TRANSFER
- Обработка успешного создания (Snackbar, навигация)
- Обновление данных после операции

**Результат:** Пользователь может фиксировать приход и расход товаров.

---

### Этап 4. Сканер штрихкодов

- Запрос разрешения на камеру в runtime
- ScannerFragment: CameraX Preview + ImageAnalysis
- ML Kit BarcodeScanner: настройка анализатора
- Поиск товара по SKU после сканирования
- Управление фонариком через `Camera.cameraControl.enableTorch()`

**Результат:** Пользователь может найти товар, отсканировав штрихкод.

---

### Этап 5. Инвентаризация

- InventoryFragment + InventoryViewModel: список с полями ввода
- InventoryAdapter с TextWatcher для real-time расчёта отклонений
- Цветовая индикация отклонений
- Сохранение результатов инвентаризации как движений типа INVENTORY

**Результат:** Пользователь может провести полную инвентаризацию склада.

---

### Этап 6. Фоновая синхронизация

- SyncWorker: `@HiltWorker`, `CoroutineWorker`, логика sync + retry
- Интеграция Hilt + WorkManager: отключение auto-init, `HiltWorkerFactory`
- SkladApplication: `Configuration.Provider`, периодическая задача (3 часа)
- Событийная синхронизация после операций (MovementFragment, InventoryFragment)
- Ограничение: выполнять только при наличии сети

**Результат:** Данные автоматически синхронизируются с сервером.

---

### Этап 7. Полировка и тестирование

- Проверка edge-to-edge на устройствах с разными типами навигации
- Проверка офлайн-режима (отключение сети)
- Проверка работы синхронизации после перезагрузки устройства
- Проверка корректности отображения данных на крайних значениях (пустой список, 0 остаток)
- Исправление найденных дефектов

**Результат:** Приложение готово к использованию.
